အရှေ့ Part 2 မှာ Coding Agent တွေကို GUI App တွေပေါ်ကနေ ဘယ်လို ကိုင်တွယ်မောင်းနှင်ရမလဲ ဆိုတာကို လေ့လာခဲ့ပြီးပြီ။ အခု **Part 3: Instructing & Delegating** ကို ရောက်တဲ့အခါ အေးဂျင့်ကို အလုပ်လွှဲအပ် ခိုင်းစေရာမှာ မင်းရဲ့ အမိန့်ပေး ညွှန်ကြားမှု (Prompting) ဟာ အဓိက အရေးအကြီးဆုံး ဖြစ်လာပါပြီ။

အေးဂျင့် တစ်ယောက်ကို ထိထိမိမိ ခိုင်းစေနိုင်ဖို့ ရိုးရိုး စာသားတွေ ပေးနေရုံနဲ့ မလုံလောက်ဘူး။ ခေတ်သစ် AI Operator တွေမှာ အစွမ်းထက်ဆုံး လျှို့ဝှက်လက်နက် ၂ ခု ရှိတယ်- အဲဒါကတော့ **Slash Commands (ဥပမာ - `/grill-me`)** နဲ့ **Specialized Skills (ဥပမာ - `ponytail`)** တို့ပဲ ဖြစ်တယ်။

ဒီ Chapter 7 မှာ အဲဒီ လက်နက် ၂ ခုက ဘာတွေလဲ၊ ဘယ်လို အလုပ်လုပ်လဲ၊ ဘယ်နေရာတွေမှာ ဘယ်လို အသေးစိတ် အသုံးချရမလဲ ဆိုတာကို လက်ဆွဲ ရှင်းပြပေးသွားမယ်။

## Slash Commands နဲ့ Skills ဆိုတာ ဘာကွာလဲ?

အသေးစိတ် မသွားခင် ဒီနှစ်ခုရဲ့ ကွာခြားချက်ကို အရင် ရှင်းရှင်းလင်းလင်း သိထားဖို့ လိုတယ်:

* **Slash Commands ( UI Shortcuts ):** Chat box ထဲမှာ `/` လေး ရိုက်ထည့်လိုက်တာနဲ့ အေးဂျင့်ကို ပုံမှန် မဟုတ်တဲ့ အထူး လုပ်ငန်းစဉ် (Special Workflow) တစ်ခု လုပ်ဆောင်ဖို့ ချက်ချင်း အမိန့်ပေးလိုက်တဲ့ ဖြတ်လမ်း ခလုတ်တွေ ဖြစ်တယ် (ဥပမာ - `/grill-me`, `/goal`, `/schedule`)။
* **Skills ( စွမ်းရည်နဲ့ စိတ်နေစိတ်ထား ):** အေးဂျင့်ကို အလုပ်တစ်ခု တာဝန်ပေးတဲ့အခါ ဘယ်လို စိတ်နေစိတ်ထားမျိုး၊ ဘယ်လို စည်းကမ်း စံနှုန်းမျိုးနဲ့ ရေးသား ဖြေရှင်းရမယ် ဆိုတာကို ပုံစံသွင်းပေးတဲ့ လမ်းညွှန်ချက် အစုအဝေး (Instructions & Mindset Folder) ဖြစ်တယ် (ဥပမာ - `ponytail` mode)။

## ၁။ `/grill-me` : အင်တာဗျူးပြီးမှ အလုပ်စမယ့် ဦးနှောက်

### ဘာလဲ?
အများစုက အိုင်ဒီယာ အကြမ်းလေး ရတာနဲ့ *"ဒီ App လေး ရေးပေးပါ"* ဆိုပြီး အေးဂျင့်ကို တန်းခိုင်းတတ်ကြတယ်။ အဲဒီအခါ အေးဂျင့်က မေဝါးတဲ့ အချက်အလက်တွေကို ကိုယ့်ဘာသာ မှန်းဆပြီး လျှောက်ရေးတော့တာပဲ။ 

**`/grill-me`** ဆိုတာ အဲဒီလို မှန်းဆ ရေးသားတာတွေကို ရပ်တန့်ပြီး၊ ကုဒ်တစ်ကြောင်းမှ မရေးခင် **အေးဂျင့်က မင်းကို ပြန်ပြီး စစ်ဆေး မေးမြန်းတဲ့ (Interview / Grill လုပ်တဲ့) Command** ဖြစ်တယ်။ သူက မင်း စဉ်းစားမထားမိတဲ့ Design ambiguities တွေ၊ Hidden assumptions တွေကို မေးခွန်းတွေ ထုတ်ပြီး အဖြေရှာပေးတယ်။

### ဘယ်လို သုံးလဲ? (Usage Guide)
Chat box ထဲမှာ ကိုယ်လုပ်ချင်တဲ့ အကြမ်းဖျဉ်း အိုင်ဒီယာရဲ့ အရှေ့မှာ `/grill-me` လို့ ခေါင်းစဉ်တပ်ပြီး ရိုက်ထည့်လိုက်ရုံပဲ:

```text
/grill-me ကျွန်တော် E-commerce website အတွက် Checkout system တစ်ခု တည်ဆောက်ချင်တယ်။
```

**အေးဂျင့်ရဲ့ တုံ့ပြန်ပုံ:**
သူက ကုဒ်တန်းမရေးဘူး။ အောက်ပါအတိုင်း မင်းကို စဉ်းစားစရာ မေးခွန်း ၃ ခု ကနေ ၅ ခုအထိ ပြင်းပြင်းထန်ထန် မေးလာလိမ့်မယ်:
1. *"Payment Gateway က ဘာသုံးမှာလဲ? Stripe လား၊ ပြည်တွင်း ကေ-ပေ (KBZPay) လား?"*
2. *"User တွေက Account ဖွင့်ပြီးမှ ဝယ်ရမှာလား၊ Guest Checkout ခွင့်ပြုမှာလား?"*
3. *"ပစ္စည်း အရေအတွက် (Inventory lock) ကို Order တင်ချိန်မှာ ဖြတ်မှာလား၊ ငွေချေပြီးမှ ဖြတ်မှာလား?"*

မင်းဘက်က အဲဒီ မေးခွန်းတွေကို ရှင်းရှင်းလင်းလင်း ပြန်ဖြေလိုက်တဲ့အခါမှ သာလျှင် အမှားအယွင်း လုံးဝ မရှိနိုင်တဲ့ တိကျတဲ့ **Implementation Plan** ကို စနစ်တကျ တည်ဆောက်ပေးပါလိမ့်မယ်။

### ဘယ်နေရာမှာ သုံးရမလဲ? (When to Use)
* ပရောဂျက် အသစ် သို့မဟုတ် Feature အသစ် တစ်ခု စတင် တည်ဆောက်တဲ့ အခါ (Requirements မကွဲပြားသေးတဲ့ အချိန်)။
* Database schema တွေ၊ Architecture တွေ အကြီးအကျယ် ပြောင်းလဲတဲ့ အခါ။
* ကိုယ့်ခေါင်းထဲမှာ အိုင်ဒီယာတော့ ရှိပေမယ့် Technical Spec စာရွက် သေချာ မရေးရသေးတဲ့ အခါ။

## ၂။ `ponytail` : ပျင်းရိပေမယ့် ထိရောက်တဲ့ Senior Dev ကြီး

### ဘာလဲ?
**`ponytail`** ဆိုတာ အတွေ့အကြုံ ရင့်ကျက်ပြီး အရာရာကို မြင်ဖူး နားဝရှိနေတဲ့ Senior Developer ကြီး တစ်ယောက်ရဲ့ စိတ်နေစိတ်ထားကို ပုံတူကူးထားတဲ့ Skill ဖြစ်တယ်။ 

သူ့ရဲ့ ဆောင်ပုဒ်က **"အကောင်းဆုံး ကုဒ်ဆိုတာ လုံးဝ မရေးရတဲ့ ကုဒ်ပဲ (The best code is the code never written)"** ဆိုတဲ့ အချက်ပါပဲ။ ဒီနေရာမှာ "ပျင်းတာ (Lazy)" ဆိုတာ ပေါ့ဆတာကို ပြောတာ မဟုတ်ဘူး။ မလိုအပ်တဲ့ Library အသစ်တွေ ထပ်မထည့်တာ၊ ရှင်းလင်းပြီး အတိုဆုံး နည်းလမ်းကို ရွေးတာ (Efficiency) ကို ဆိုလိုတာ ဖြစ်တယ်။

### Ponytail ရဲ့ ဆုံးဖြတ်ချက် လှေကားထစ်များ (The Lazy Ladder)
ကုဒ်တစ်ကြောင်း မရေးခင် `ponytail` ဟာ အောက်ပါ လှေကားထစ် ၇ ထစ်အတိုင်း အမြဲ စစ်ဆေးတယ်:
1. **YAGNI (You Aren't Gonna Need It):** ဒီ feature ကို တကယ်ပဲ အခု ချက်ချင်း လိုအပ်နေတာ လား? မလိုရင် လုံးဝ မရေးဘူး။
2. **Existing Code:** ကိုယ့် Codebase ထဲမှာ ဒီလုပ်ဆောင်ချက်မျိုး လုပ်ပေးတဲ့ Helper/Util ရှိပြီးသားလား? ရှိရင် ပြန်သုံးမယ်။
3. **Standard Library:** JavaScript/Python ရဲ့ Built-in Standard Library ထဲမှာ ပါပြီးသားလား? ပါရင် npm dependency အသစ် လုံးဝ မထည့်ဘူး။
4. **Native Feature:** OS သို့မဟုတ် Platform ရဲ့ Native feature နဲ့ ရနိုင်လား? ရရင် သုံးမယ်။
5. **Installed Dependency:** လက်ရှိ တပ်ဆင်ထားပြီးသား Library တွေနဲ့ ဖြေရှင်းလို့ ရလား?
6. **One-liner:** စာကြောင်း ၅၀ ရေးမယ့်အစား ၁ ကြောင်းတည်းနဲ့ ပြီးအောင် ရေးလို့ ရမလား?
7. **Minimal Code:** အပေါ်က နည်းလမ်းတွေ အကုန် မရတော့မှသာ အလုပ်ဖြစ်မယ့် အတိုဆုံး ကုဒ်ကို ရေးတယ်။

### ဘယ်လို သုံးလဲ? (Usage Guide)
အေးဂျင့်ကို ခိုင်းတဲ့အခါ Prompt ထဲမှာ `ponytail`, `be lazy`, `lazy mode`, `simplest solution` သို့မဟုတ် `yagni` ဆိုတဲ့ စကားလုံးတွေ ထည့်သုံးလိုက်ရုံနဲ့ ဒီ Skill က အလိုအလျောက် အလုပ်လုပ်တယ်:

```text
ဒီ User list ကို နေ့စွဲအလိုက် Sort စီပေးပါ။ ponytail mode နဲ့ အရှင်းဆုံးနဲ့ အတိုဆုံး နည်းလမ်းကိုပဲ သုံးပါ။
```

**Intensity Levels (အပြင်းအဆော့ အဆင့်များ):**
* `ponytail lite` : အခြေခံ ရှင်းလင်းမှုလောက်ပဲ လုပ်ဆောင်ပြီး ထင်ရှားတဲ့ ဘလော့ (Bloat) တွေကို ဖယ်ရှားတယ်။
* `ponytail full` (Default) : လှေကားထစ် ၇ ထစ်လုံးကို တိတိကျကျ စစ်ဆေးပြီး အလွန် ထိရောက်တဲ့ ကုဒ်ကို ရေးတယ်။
* `ponytail ultra` : အစွန်းရောက် ရှင်းလင်းရေး။ မလိုအပ်တဲ့ Abstraction တွေ အားလုံးကို အပြတ် ရှင်းလင်းပြီး အတိုဆုံး ဖြစ်အောင် ညှစ်ထုတ်တယ်။

**ဆက်စပ်နေတဲ့ Ponytail Commands များ:**
* `/ponytail-review` : ကိုယ့်ရဲ့ Code တွေကို စစ်ဆေးတဲ့အခါ Over-engineering ဖြစ်နေတာတွေ၊ မလိုအပ်တဲ့ Dependency တွေကို ရှာဖွေပြီး ဖျက်ပစ်ဖို့ အကြံပေးတဲ့ Reviewer။
* `/ponytail-audit` : တစ်ပရောဂျက်လုံး (Whole-repo) ကို Scan ဖတ်ပြီး ဘာတွေ ဖျက်ပစ်လို့ ရလဲ၊ ဘာတွေကို ရှင်းလင်းလို့ ရလဲ ဆိုတာ ရာခိုင်နှုန်းနဲ့ ထုတ်ပြတဲ့ Audit tool။
* `/ponytail-debt` : Codebase ထဲမှာ ယာယီ ဖြတ်လမ်းနည်း သုံးထားလို့ ချန်ခဲ့တဲ့ `ponytail:` comment တွေကို စုစည်းပြီး Technical Debt စာရင်း ထုတ်ပေးတဲ့ tool။

### ဘယ်နေရာမှာ သုံးရမလဲ? (When to Use)
* အေးဂျင့်က အလုပ် သေးသေးလေး တစ်ခုအတွက် စာကြောင်းရေ ရာချီရှည်တဲ့ Boilerplate ကုဒ်တွေ လျှောက်ရေးလာတဲ့ အခါ။
* `npm install` တွေ လျှောက်လုပ်ပြီး Package အသစ်တွေ အလွယ်တကူ ထည့်ချင်နေတဲ့ အခါ။
* Pull Request (PR) မတင်ခင် ကိုယ့်ကုဒ်တွေ ရှုပ်ထွေးနေလား ဆိုတာကို သန့်စင် ရှင်းလင်းချင်တဲ့ အခါ။

## လက်တွေ့ ဥပမာ နှိုင်းယှဉ်ချက် (Vibe Coder vs. Ponytail Operator)

User object တွေ ပါဝင်တဲ့ Array တစ်ခုကို ရက်စွဲ (Date) အလိုက် စီရမယ့် အလုပ်တစ်ခုကို ကြည့်ရအောင်:

**Vibe Coder ရဲ့ နည်းလမ်း (No Ponytail):**
အေးဂျင့်ကို စည်းကမ်း မလုံခြုံဘဲ ခိုင်းလိုက်တဲ့အခါ သူက `moment.js` သို့မဟုတ် `lodash` လို Library ကြီးကို install လုပ်ခိုင်းပြီး၊ Custom Sorting Interface တွေ၊ Error wrapper class တွေနဲ့ စာကြောင်းရေ ၃၀ လောက်ရှိတဲ့ ကုဒ်အရှည်ကြီးကို ရေးချပေးလိမ့်မယ်။

**Ponytail Operator ရဲ့ နည်းလမ်း (With Ponytail):**
`ponytail` mode နဲ့ ခိုင်းလိုက်တဲ့အခါ သူက Standard Library ထဲက `Array.prototype.sort()` ကို တန်းမြင်ပြီး အောက်ပါအတိုင်း ၁ ကြောင်းတည်းသော ရှင်းလင်းတဲ့ Native ကုဒ်ကို ရေးသားပေးတယ်:

```javascript
// ponytail: Native JS sort by timestamp (O(n log n))
const sortedUsers = users.sort((a, b) => new Date(b.date) - new Date(a.date));
```
**ရလဒ်:** Dependency အသစ် ထည့်စရာ မလိုဘူး၊ Code base ပေါ့ပါးသွားတယ်၊ မြန်ဆန်ပြီး ထိန်းသိမ်းရ လွယ်ကူသွားတယ်။

## သတိကြီးစွာ ထိန်းကျောင်းရမယ့်အချက်တွေ (Risks & Caveats)

<Warning>
**၁။ Over-grilling (မေးခွန်းတွေ သံသရာပတ်ပြီး အလုပ်မစနိုင်တာ):**
`/grill-me` က အစီအစဉ်ချဖို့ ကောင်းပေမယ့် စာကြောင်း တစ်ကြောင်းလောက် ပြင်ရမယ့် Bug fix သေးသေးလေးတွေအတွက် သွားမသုံးပါနဲ့။ မကျေကွဲတဲ့ မေးခွန်း ၅၀ လောက် ထိုင်ဖြေနေရရင် အချိန်ကုန်ပြီး အလုပ်မတွင်တာမျိုး ဖြစ်သွားတတ်တယ်။

**၂။ Too Lazy to Check (Test မစမ်းဘဲ ဖြတ်လမ်းလိုက်တာ):**
`ponytail` က ကုဒ်တိုတာကို နှစ်သက်ပေမယ့် **Security (လုံခြုံရေး)**၊ **Input Validation** နဲ့ **Error Handling** တွေမှာ လုံးဝ (လုံးဝ) မပျင်းပါဘူး။ ကုဒ်တိုချင်ဇောနဲ့ မဖြစ်မနေ စစ်ဆေးရမယ့် Security check တွေကို အေးဂျင့်က ဖြတ်ဖျက်မသွားအောင် Operator ဘက်က သတိထား စောင့်ကြည့်ရမယ်။
</Warning>
